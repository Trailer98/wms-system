# WMS RAG 知识库 —— 设计候选项（未实现，不导入知识库）

本文件记录当前代码扫描中发现"设计上合理、但当前代码尚未实现"的规则构想，主要来自本次修复过程中从既有知识文档里剥离出来的财务事件相关描述。这里的内容**不是**可以直接导入 `knowledge_document` 的正式知识，仅作为后续如果要实现相关功能时的参考起点。

结构化版本同时保存在 `docs/generated/wms-rag-knowledge-rules-v1-fixed.json` 的 `designCandidates` 数组中（`status` 字段固定为 `DRAFT_NOT_FOR_IMPORT`，不是系统里任何真实的知识状态值，避免被误当作可导入数据）。

## 为什么财务事件被列为设计候选项而不是已实现规则

本次代码扫描明确确认：当前代码库中**没有 `finance_event` 表，没有对应的 Service，没有对应的 Controller 或接口**。`StockAdjustOrderService.java` 的源码注释中原文写明：

> "there is no finance-event service in this codebase yet"

`EnterpriseYearDemoDataIntegrityChecker` 中也有一条专门的检查项 `noFinanceEventTableCheck()`，用来确认 `finance_event` 表确实不存在。

既有知识文档中曾出现"库存调整会生成财务影响事件""正常区转异常区/异常区恢复通常会（或不会）生成财务损益事件"这类表述，这些描述把"财务事件"当作系统里已经存在、只是触发条件不同的功能来讲解，这是不准确的——正确的说法应该是"当前系统完全没有财务事件功能"，而不是"某些操作触发、某些操作不触发"。这些错误表述已经在 `wms-rag-knowledge-rules-v1-fixed.json` 的 `fixRules` 中被替换为如实说明"当前未实现"的版本。

## 设计候选项 1：库存操作触发财务事件规则（草案）

- **docCode（草案）**：`FINANCE_EVENT_TRIGGER_RULE_DRAFT`
- **模块**：FINANCE
- **实现状态**：未实现（NOT_IMPLEMENTED）

如果未来要实现库存操作关联财务事件的功能，需要评审的问题包括：

1. 哪些库存变化类型应该触发财务事件？候选：库存调增、库存调减、盘盈、盘亏、报废（异常区库存最终注销）。哪些不应该触发？候选：出库锁库、出库取消解锁、正常区/异常区之间的库存转移（因为只是状态和位置变化，资产总量不变）。
2. 财务事件需要记录哪些字段？例如涉及金额、单位成本、来源单据类型和单号、触发的库存流水 ID 等。
3. 财务事件和现有的 `stock_movement`（库存流水）表是什么关系？是在流水表基础上派生的只读视图，还是独立的新表、需要单独维护？
4. 是否需要人工确认财务事件才能生效？由谁负责确认？确认后是否允许撤销或红冲？

## 设计候选项 2：财务事件状态规则（草案）

- **docCode（草案）**：`FINANCE_EVENT_STATUS_RULE_DRAFT`
- **模块**：FINANCE
- **实现状态**：未实现（NOT_IMPLEMENTED）

如果未来要实现，需要定义财务事件自身的状态枚举（例如：待确认、已确认、已生成凭证、已作废）及其之间的流转规则，并明确谁有权限执行每一步状态变化。

## 使用建议

1. 这两条设计候选项**不应该**被导入 `knowledge_document`，`status` 也不应设置为 `ENABLED`。
2. 如果后续团队决定实现财务事件功能，应该先完成代码实现，再基于真实代码重新编写正式的知识规则（走本次同样的"先扫描代码、再生成知识"流程），而不是直接把这份设计草案原样导入。
3. 在此之前，RAG 问答遇到"库存操作会不会影响财务"这类问题，应当如实回答"当前系统未实现财务事件功能"，参见 `wms-rag-knowledge-rules-v1-fixed.json` 中对应 fixRules 修正后的内容。
